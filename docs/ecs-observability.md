# ECS 观测设计

## 背景

数据中台及相关系统通常由多台 ECS、服务组件和对外接口共同组成。ECS 是最底层的运行基础，如果主机资源异常，服务组件和对外服务往往会出现连锁问题。

本主题先建设 ECS 层面的统一观测视图，用于快速判断一批 ECS 的整体健康情况，并支持在发现异常后下钻到单台 ECS 排查。

## 设计目标

ECS dashboard 的目标分为三层：

1. 总览：第一屏直接判断是否有问题。正常情况下只看总览即可，不需要继续向下排查。
2. 大概明细：当总览出现风险时，用系统级趋势和 Top 视图判断问题类型和影响范围。
3. 详细明细：选择具体 IP 后查看单台 ECS 的 CPU、内存、磁盘、网络、负载、进程和基础信息，用于定位原因。

当前 dashboard 对应文件：

- `grafana/dashboards/local-node-exporter.json`：数据中台 ECS 监控总览
- `grafana/dashboards/ecs-node-detail.json`：数据中台 ECS 单机明细
- `prometheus/prometheus.yml`

## 当前数据模型

当前用本地 `node-exporter` mock 10 台 ECS。Prometheus 会多次采集同一个 `node-exporter:9100`，并给每个采集目标打不同标签。

最终 dashboard 以 `参考/数据中台-ECS监控-1778728407266.json` 中的内网标签为基准：

- `vendor`：云厂商
- `account`：账户
- `group`：分组，当前为 `数据中台`
- `name`：ECS 展示名称
- `instance`：IP
- `iid`：实例 ID

真实接入 ECS 时，应优先保留 `vendor/account/group/name/instance/iid` 的标签语义。详细标签和面板口径见 [数据中台 ECS Dashboard 标签与面板口径](ecs-dashboard-labels.md)。

## 总览设计

总览区放在最上方，主要用于“一眼判断是否有问题”。

顶部状态卡片：

- `系统`：Grafana 顶部变量，来自 Prometheus 的 `group` 标签，所有系统共用一份总览。
- `需关注 / 已接入 ECS`：在一个状态卡里展示健康值低于阈值的 ECS 数量和已接入 ECS 总数，两个值都按整数显示。
- `最高 CPU`：展示当前最高 CPU 使用率，使用仪表盘展示百分比水位。
- `最高内存`：展示当前最高内存使用率，使用仪表盘展示百分比水位。
- `最高磁盘繁忙度`：展示当前最高 I/O Util，使用仪表盘展示百分比水位。
- `最高负载 5m`：展示当前系统分组内单台 ECS 的最高 5 分钟负载。

资源风险总览表参考运维平台资源表风格，字段聚焦是否接近阈值和是否需要处理：

- 名称
- IP
- 实例 ID
- 在线
- 健康值
- CPU 使用率
- 内存使用率
- Swap 使用率
- 分区最高使用率
- 最小可用空间
- IOutil 使用率
- 5 分钟负载
- 阻塞进程
- 磁盘读取
- 磁盘写入
- 下载带宽
- 上传带宽
- TCP 连接数
- TIME_WAIT 连接数
- 启动天数

关键指标使用颜色背景表达风险：绿色表示正常，橙色表示需要关注，红色表示异常或高风险。

总览 dashboard 删除了 CPU 使用率及以下的趋势面板，只保留第一屏统计和资源风险总览表。需要看单机趋势时，从表格 IP 跳转到单机明细。

## 异常判定

当前异常主要围绕 ECS 是否在线以及关键资源是否过高。

### 在线异常

在线状态来自 Prometheus 的 `up` 指标：

```promql
up{job="node", group="数据中台"}
```

判定规则：

- `up == 1`：ECS exporter 可采集，认为在线。
- `up == 0`：ECS exporter 不可采集，认为异常。

异常 ECS 数量：

```promql
count(up{job="node", group="数据中台"} == 0) or vector(0)
```

这里使用 `or vector(0)`，保证没有异常时也返回 `0`，Grafana 再将 `0` 映射为 `无`。

### 资源风险

当前资源风险主要观察：

- CPU 使用率
- 内存使用率
- IOutil 使用率
- 负载
- 阻塞进程
- 磁盘读写速率

建议阈值：

- CPU 使用率：`70%` 以上关注，`90%` 以上高风险。
- 内存使用率：`70%` 以上关注，`90%` 以上高风险。
- IOutil 使用率：`60%` 以上关注，`85%` 以上高风险。
- 负载：需要结合 CPU 核数判断，当前先用颜色提示趋势。
- 阻塞进程：持续大于 `0` 需要关注。

这些阈值是初始经验值，后续应根据真实 ECS 规格和业务压力调整。

## 健康度计算

健康值用于把多个资源风险压缩成一个可快速扫描的分数。当前计算思路是：取 CPU、内存、IOutil 三者中最高的风险值，然后从 100 分里扣分。

核心逻辑：

```promql
100 - clamp_max(max_resource_usage, 100) / 10
```

其中 `max_resource_usage` 是当前 ECS 的 CPU 使用率、内存使用率、IOutil 使用率中的最大值。

示例：

- 最大资源使用率为 `20%`，健康值约为 `98`。
- 最大资源使用率为 `70%`，健康值约为 `93`。
- 最大资源使用率为 `90%`，健康值约为 `91`。
- 最大资源使用率为 `100%`，健康值为 `90`。

当前健康值不是严格的故障评分，而是总览表中的快速排序和颜色参考。它的作用是帮助运维先看到“哪台 ECS 更值得关注”。后续可以把在线状态、阻塞进程、磁盘写入异常、网络错误率等纳入健康度，使分数更接近真实风险。

## 详细明细联动

详细明细已经拆到独立 dashboard。总览表中的 IP 列会跳转到 `数据中台 ECS 单机明细`，并通过 `var-instance` 传入具体 IP。单机明细中的所有查询都使用：

```promql
instance=~"$instance"
```

详细区标题会显示：

```text
数据中台 ECS 单机明细：$show_name / $instance
```

这样排障时可以明确当前查看的是哪台 ECS。

详细明细包含：

- 在线状态
- CPU 使用率
- 内存使用率
- 磁盘繁忙度
- 1 分钟负载
- CPU 使用率明细
- 内存容量与使用率
- 分区容量明细
- 系统负载与进程状态
- 文件描述符与上下文切换
- 磁盘 I/O 明细
- 网络与连接明细

磁盘 I/O 和网络连接不再作为独立分区展示，它们都属于所选 ECS 的单机排障内容。

## 后续演进

ECS 主题后续建议补充：

- 真实 ECS node-exporter 自动发现。
- ECS 所属系统、服务角色、负责人等标签。
- 文件系统使用率指标。如果真实环境暴露 `node_filesystem_*`，可以重新加入分区使用率。
- 网络错误、丢包、TCP 重传等网络健康指标。
- 基于真实故障经验调整健康度公式。
- 告警规则与 dashboard 阈值保持一致。
